IDS ALERT CLASSIFIER — THESIS DEVELOPMENT NOTES
Fine-tuning Qwen2.5-1.5B-Instruct for network intrusion detection

PROJECT OVERVIEW
----------------

This project serves a dual purpose:

1. MASTER'S THESIS
   Final thesis submission for a cybersecurity master's degree. The project
   investigates the viability of fine-tuning a small instruction-tuned LLM
   (Qwen2.5-1.5B-Instruct) for real-time network intrusion detection using
   Zeek conn.log data. Key research questions include:
     - Can a fine-tuned LLM match or exceed classical ML accuracy on IDS tasks?
     - Does multi-source heterogeneous training data improve generalisation to
       real-world traffic compared to lab-only datasets?
     - Does the explainability of LLM output (VERDICT + REASON) provide
       practical value over black-box classifiers in a SOC context?

2. REAL DEPLOYMENT TARGET
   Beyond the thesis, the goal is a working IDS classifier that runs locally
   on consumer hardware (RTX 3070, 8 GB VRAM) via Ollama, classifying live
   Zeek conn.log and weird.log entries in real time. The system must be:
     - Fast enough for interactive use on local hardware
     - Accurate enough to be trusted (low false positive rate on real traffic)
     - Explainable — every verdict comes with a human-readable reason

--------------------------------------------------------------------------------

This file is a running log of design decisions, version changes, findings, and
reasoning. It is intended to support the final master's thesis write-up.
Never delete — append new entries as the project evolves.

================================================================================
MODEL CHOICE
================================================================================

Base model: Qwen/Qwen2.5-1.5B-Instruct
Quantization: 4-bit NF4 (QLoRA) via BitsAndBytes, bf16 compute dtype

Why an LLM over classical ML or BERT?
- Classical ML (Random Forest, XGBoost) and BERT achieve high accuracy on
  CICIDS2017 but are known to overfit its statistical artifacts — this is
  well-documented in the literature. They generalise poorly to out-of-
  distribution traffic.
- BERT fine-tuned on network flows would match accuracy but produces no
  explanation — it is a black box requiring a separate explainability layer
  (LIME, SHAP) to be useful to a SOC analyst.
- The LLM approach produces a structured VERDICT + REASON for every alert.
  Explainability is a hard requirement in real SOC environments. A Random
  Forest that outputs "ATTACK: 0.94" is not actionable. A response of
  "VERDICT: ATTACK / REASON: Asymmetric byte distribution and connection
  state indicate hostile intent." is.
- The REASON field is the key academic and practical differentiator.

Why Qwen2.5-1.5B specifically?
- Small enough to run on consumer hardware (RTX 3070, 8 GB VRAM) at inference
  via 4-bit loading. Deployment cost is zero beyond the local machine.
- Instruction-tuned base makes it respond well to the structured prompt format
  without requiring large amounts of data to learn the output schema.
- QLoRA means only a small adapter (~20 MB) is trained and stored — the frozen
  4-bit base model is hardware-agnostic.

Known limitation to address in thesis:
- Binary classification (ATTACK / FALSE POSITIVE) is a simplification. Real
  IDS needs multi-class attack categorisation. This is a scoped first step.
- A potential extension is a third class (UNKNOWN / low-confidence) with
  calibration metrics to model uncertainty.

================================================================================
HOW LoRA AND QLoRA WORK
================================================================================

--- The problem LoRA solves ---

A full fine-tune of Qwen2.5-1.5B means updating all 1.5 billion parameters
via gradient descent. Each parameter needs:
  - The parameter itself    (bf16 = 2 bytes)
  - Its gradient            (2 bytes)
  - Two optimizer momentum values (8 bytes each in fp32)

That is ~18 GB just for optimizer states on a 1.5B model. Impossible on
consumer hardware, and wasteful — you are retraining a model that already
knows language, reasoning, and structure just to teach it one narrow task.

--- The key insight ---

When you fine-tune a large model on a specific task, the CHANGE in each weight
matrix turns out to be low-rank. This was observed empirically by the LoRA
authors (Hu et al., 2021).

Low-rank means: a 1536×1536 matrix has 2.3M independent numbers. But if the
useful change to that matrix can be described by combining just 16 "directions",
you do not need 2.3M numbers. You can store two small matrices that, when
multiplied together, reconstruct that change. That product is a rank-16
approximation of the full update.

This is the bet LoRA makes: the task-specific update lives in a low-dimensional
subspace of the weight space.

--- The mechanism ---

For each target weight matrix W (e.g. query projection, shape 1536×1536):

  Freeze W completely. Add two new matrices alongside it:
    A: shape (1536 × 16)  — projects DOWN into rank space
    B: shape (16  × 1536) — projects UP back to original dimension

  Forward pass becomes:
    output = W · x  +  (B · A) · x
             ^^^^^^     ^^^^^^^^^^^
          frozen base   learned correction

  At the start of training, B is initialised to zeros, so B·A = 0 and the
  adapter adds nothing. The model starts identical to the base. A and B
  learn together during training to produce corrections that push outputs
  toward the task objective.

  The scaling factor lora_alpha / r (32/16 = 2.0 in v6) controls the
  magnitude of the correction.

--- What "rank" means intuitively ---

Imagine describing the change from a neutral face to a smiling face. A full
1024×1024 image has 1M numbers, but the change only involves a few directions:
lip corners up, cheeks raised, etc. Maybe 10-20 directions are enough.

Rank is the number of those directions. r=1 is extremely constrained.
r=16 gives 16 directions of change. r=full_dimension recovers the full
fine-tune. For binary classification of structured network flows, r=16
is generous — the task is narrow and consistent.

--- What gets trained vs frozen ---

  Frozen (never updated, no gradient, no optimizer state):
    - All original W matrices (attention projections, MLP weights, etc.)
    - Embedding layers
    - Layer norms

  Trained (gradients computed, optimizer states held):
    - A and B matrices for each targeted layer
    - v6 targets: q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj
    - 7 modules × 28 layers × 2 matrices = 392 small matrices
    - Total trainable parameters: ~688k out of 1.5B (0.046%)
    - Adapter size on disk: ~20 MB

--- What QLoRA adds ---

LoRA alone still requires the base model in full precision (~3 GB in bf16).
QLoRA loads the base model in 4-bit NF4 quantization (~900 MB instead of 3 GB).

The 4-bit weights are never updated. During the forward pass each layer is
dequantized to bf16 on the fly for computation, then discarded. The LoRA
matrices A and B remain in full bf16 throughout. Gradients only flow into
A and B — never into the frozen quantized weights.

--- What "activations" are and why they matter for VRAM ---

Activations are the intermediate tensors produced at every step of the forward
pass. Every layer takes the previous layer's output as input and produces a
new output tensor — both are activations.

During training, the backward pass computes gradients by working backwards
through the network. To do this it needs the intermediate values from the
forward pass. By default, every intermediate tensor from every layer is kept
in VRAM until the backward pass finishes.

Activation memory scales with:
  - Batch size: 22 samples processed simultaneously
  - Sequence length: up to 512 tokens (~100 in practice for these prompts)
  - Hidden dimension: 1536 numbers per token per layer
  - 28 layers × multiple intermediate tensors per layer

This is why gradient_checkpointing=True is required even on 32 GB VRAM.
Instead of storing all activations, it keeps only activations at a subset
of checkpoint layers and recomputes the rest on demand during the backward
pass. This trades ~20% extra compute for a large reduction in VRAM usage.
Disabling it caused VRAM to spike from ~19 GB to ~31 GB (96% of 32 GB).

--- VRAM breakdown for v6 on RTX 5090 ---

  4-bit base model:          ~900 MB
  LoRA adapter (bf16):        ~20 MB
  Activations (batch=22):  ~10-12 GB  (dominant cost, reduced by grad ckpt)
  Optimizer states:            ~5 MB  (paged to CPU RAM under pressure)
  CUDA / framework overhead:  ~1-2 GB
  Total observed:            ~14-19 GB

--- What the trained adapter captures ---

After training on 330k labelled flows, A and B have learned corrections that
at each layer push the model's internal representations in task-relevant
directions:

  - Short-duration, high-packet-count, asymmetric flows → ATTACK region
  - Clean SF-state, balanced byte-ratio flows → FALSE POSITIVE region
  - Output format: VERDICT: X / REASON: Y

The base model provides the representational space (language understanding,
structured output, reasoning). The adapter provides the directions — which
way to move in that space for this specific task.

The correction is small in magnitude but applied at every layer of every
forward pass. 28 layers of small nudges add up to meaningfully different
final behaviour.

  Input prompt (network flow text)
          │
          ▼
  [Embedding layer — frozen]
          │
          ▼  (repeated × 28 layers)
  ┌──────────────────────────────────┐
  │  W_frozen · x                    │  ← original Qwen knowledge
  │        +                         │
  │  B · A · x  (scaled by α/r=2.0) │  ← IDS correction from training data
  └──────────────────────────────────┘
          │
          ▼
  [Output head — frozen]
          │
          ▼
  "VERDICT: ATTACK\nREASON: ..."

================================================================================
PROMPT FORMAT (all versions)
================================================================================

System prompt:
  "You are a network security analyst. Always respond with
   VERDICT: <ATTACK or FALSE POSITIVE> on the first line,
   followed by REASON: <brief explanation>."

Input: 10 Zeek-native fields — 7 base + 3 derived:
  Proto, Duration (s), Orig Packets, Resp Packets, Orig Bytes, Resp Bytes,
  Conn State, Bytes/sec, Orig Bytes/Pkt, Resp Bytes/Pkt

Derived fields propagate N/A when base fields are missing (e.g. S0
connections with no byte counts). This mirrors real inference behaviour
where Zeek logs frequently contain dash placeholders for unset fields.

Expected output:
  VERDICT: ATTACK
  REASON: <brief explanation>

The prompt format was kept identical across training, benchmarking, and live
classification (classify_conn_log.py, classify_weird_log.py) to avoid any
train/inference mismatch.

v4 — COMPLETED
--------------------------------------------------------------------------------
Dataset:    4 sources — IoT-23, CTU-13, UNSW-NB15, CICIDS2017
            ~300k samples total
Prompt:     Zeek-native 10-field prompt (same as v3 but redesigned)
Accuracy:   ~82% on CICIDS2017 benchmark
LoRA:       r=8, lora_alpha=16
Epochs:     1
Hardware:   Local RTX 3070

Why the step down from 89% to 82%:
- The new prompt uses Zeek-native fields only. CICIDS2017 has no conn_state
  (set to "-") and uses numeric protocol codes ("6" instead of "tcp").
  The model saw more diverse, harder data — the accuracy drop is expected
  and acceptable given the gain in real-world generalisability.

Critical finding — real-world deployment failure:
- When run against a real Zeek conn.log (classify_conn_log.py), v4 produced
  an approximately 95% FALSE POSITIVE rate. Nearly all legitimate traffic
  was flagged as ATTACK.
- Root cause: the training data (IoT-23, CTU-13, UNSW-NB15, CICIDS2017) is
  all lab-generated or synthetic. The model had never seen realistic benign
  traffic patterns from actual user browsing and enterprise activity.
  It learned to classify "unusual-looking" flows as attacks, and real traffic
  contains many flows that look unusual by lab standards (e.g. short S0
  connections, zero-byte responses, varied protocol behaviour).
- The dash/N/A propagation was also broken in v4: missing fields (Zeek
  uses "-" as a placeholder) were falling through to 0.0 in derived fields
  (Bytes/sec, Orig Bytes/Pkt, Resp Bytes/Pkt), creating a numeric signal
  that was inconsistent between training and inference.

v6 — COMPLETED (training started 2026-03-28)
--------------------------------------------------------------------------------
Dataset:    6 sources — IoT-23, CTU-13, UNSW-NB15, CICIDS2017,
            UWF-ZeekData24, CTU-Normal
            ~330k samples (180k ATTACK / 150k FALSE POSITIVE)
LoRA:       r=16, lora_alpha=32 (doubled adapter capacity vs v4)
Epochs:     3 (vs 1 in v4)
Hardware:   RunPod RTX 5090 (32 GB VRAM), $0.90/hr
Estimated training cost: ~$5-6

Key changes from v4 and why:

1. Added UWF-ZeekData24 (real Zeek conn.log, University of West Florida
   cyber range, MITRE ATT&CK labelled):
   - Real network traffic from a university cyber range, not synthetic.
   - Zeek-native column names — minimal preprocessing needed.
   - Directly addresses the real-world FP problem from v4.

2. Added CTU-Normal (benign-only Zeek conn.log from Stratosphere Lab,
   CTU-Normal-20 through 32):
   - Real user browsing and background traffic, no attacks.
   - Provides the model with realistic benign patterns it had never seen.
   - All 80k samples labelled FALSE POSITIVE.
   - Directly addresses the real-world FP problem from v4.

3. Fixed N/A propagation (prompt_utils.py):
   - In v4, missing fields ("-", empty string) fell through to 0.0 in
     derived calculations. Now _is_na() detects unset fields and derived
     fields (Bytes/sec, Bytes/Pkt) propagate as "N/A" rather than "0.0".
   - Training data and inference prompts now match exactly for flows with
     missing fields (e.g. S0 connection state with no byte counts).

4. Fixed IoT-23 dash handling:
   - v4 used len(parts) < 22 to skip malformed lines. The correct check
     is len(parts) < 21 — the last IoT-23 field bundles tunnel_parents,
     label, and detailed-label as space-separated sub-tokens within a
     single tab field. The off-by-one caused valid labelled lines to be
     silently dropped.

5. Shared prompt_utils.py (single source of truth):
   - In v4, build_prompt() was duplicated across preprocess_zeek.py,
     benchmark.py, and classify_conn_log.py. Any divergence would create
     a train/inference mismatch. Now all scripts import from one module.

6. Increased LoRA rank: r=8 → r=16, lora_alpha=16 → 32:
   - Doubles adapter capacity. The harder attack types (web attacks,
     infiltration) showed lower per-class accuracy in v4. More expressive
     adapters should help without significant VRAM impact (~20 MB more).

7. Increased epochs: 1 → 3 with cosine_with_restarts (3 cycles):
   - v4 trained for only 1 epoch — likely underfit given the dataset size
     and task complexity. 3 epochs with 1 restart per epoch allows the
     model to consolidate learning while the cosine schedule prevents
     overfitting at the end of each cycle.

8. Moved training to RunPod (RTX 5090):
   - Local RTX 3070 (8 GB VRAM) cannot fit batch=12 with the full v6
     config. RunPod provides on-demand access to 32 GB VRAM at $0.90/hr.
   - Training cost per run is ~$5-6, making iteration affordable.

Training configuration (v6 final):
  per_device_train_batch_size = 22
  gradient_accumulation_steps = 1    (effective batch = 22)
  optim = "paged_adamw_8bit"         (pages optimizer states to CPU RAM
                                      under pressure — keeps VRAM ~19 GB)
  gradient_checkpointing = True      (required even on 32 GB — disabling
                                      it caused VRAM to spike to 31/32 GB)
  bf16 = True
  num_train_epochs = 3
  learning_rate = 2e-4
  lr_scheduler_type = "cosine_with_restarts"
  num_cycles = 3                     (1 restart per epoch)
  warmup_ratio = 0.03
  weight_decay = 0.01
  max_length = 512                   (prompts are ~80-100 tokens; no
                                      truncation occurs in practice)
  eval_strategy = "epoch"
  load_best_model_at_end = True      (selects checkpoint with lowest
                                      eval_loss across 3 epochs)

Notes on training configuration decisions:
- paged_adamw_8bit was chosen over adamw_torch_fused: switching to fused
  caused VRAM to jump from ~19 GB to ~31 GB (close to OOM on 32 GB).
  paged offloads optimizer states to CPU under memory pressure, which is
  why VRAM stays low despite a large dataset.
- gradient_checkpointing=True was initially disabled thinking 32 GB was
  sufficient. Disabling it caused VRAM to spike to 31/32 GB (96%).
  Re-enabled — it trades ~20% compute speed for significant VRAM saving
  by recomputing activations during the backward pass instead of storing them.
- Batch size was tuned from 12 → 22 after observing ~13 GB VRAM usage at
  batch=12, leaving substantial headroom. Batch=22 estimated at ~18 GB,
  giving a safe margin while improving throughput.

================================================================================
V6 ROOT CAUSE ANALYSIS
================================================================================

The failure of v6 on real-world data was diagnosed by inspecting the conn_state
distribution per (source, class) in the benchmark dataset.

Problem 1 — IoT-23 benign dominates the benign class with S0 UDP (89% S0)
  IoT-23 benign: 268/300 S0, 249/300 UDP
  At 80k samples it is the largest single benign source. The model learned
  "S0 UDP = benign". CTU-Normal (94% SF) and UWF benign (76% SF) do not
  match this pattern, so the model flags them as suspicious.

  Why IoT-23 benign is so S0-heavy: IoT devices (sensors, smart home) mostly
  send unreplied packets — NTP probes, DNS queries to unreachable servers,
  SSDP/mDNS broadcasts. These produce many S0 flows. Real user PC browsing
  produces mostly SF flows (completed TCP connections).

Problem 2 — UWF Credential Access attacks are unlearnable
  UWF attacks: 100% SF, 100% TCP, mean duration ~0.02s
  These are credential brute-force flows that look identical to normal short
  web connections. Training recall is 2% — the model cannot learn to
  distinguish them from benign traffic using flow-level features alone.
  Worse, training on them teaches "short SF TCP = ATTACK", which is exactly
  the pattern of legitimate API calls, DB queries, and HTTPS connections.

Problem 3 — CTU-13 binetflow uses Argus state notation, not Zeek
  CTU-13 attack conn_states: 92% "INT" (Argus mid-flow state)
  CTU-13 benign conn_states: 63% "CON", 16% "FSPA_FSPA" (Argus completed)
  Production Zeek logs never produce INT, CON, FSPA_FSPA, etc. The model
  was trained on a conn_state vocabulary that does not exist at inference time.

Problem 4 — CICIDS2017 provides 80k samples with two critical features missing
  All CICIDS2017 samples: conn_state="-", proto="unknown"
  CICFlowMeter (the tool used to generate CICIDS2017) has no concept of
  Zeek conn_state. Protocol is numeric (6, 17) and the loader maps unknown
  values to "unknown". 80k training samples where both the most discriminative
  features are absent adds noise and teaches the model to classify without them.
  CICIDS2017 also has documented label quality issues (~10-15% mislabeled flows).

================================================================================
V7 — CHANGES AND RATIONALE
================================================================================

v7 directly addresses the four root causes identified above. Dataset: ~360k
samples (120k attacks, 240k benign). 2:1 benign:attack ratio.

Training: RunPod RTX 3090 (24 GB VRAM) recommended, or local RTX 3070 with
TRAINING_FACTOR=0.1 for a fast validation run before committing to full cost.

Change 1 — IoT-23 benign cap: 80k → 20k
  Rationale: At 80k, IoT-23 benign (89% S0 UDP) dominated the benign class
  and created a "S0 = benign" bias. Reducing to 20k limits its influence
  while retaining sufficient S0 UDP benign examples for that pattern.

Change 2 — Remove UWF attacks entirely (keep UWF benign)
  Rationale: All UWF attacks are Credential Access (100% SF TCP, ~0.02s
  duration, 2% recall). They are unlearnable from flow features and actively
  teach a harmful pattern (short SF TCP = ATTACK). UWF benign (real Zeek,
  mixed SF/S0 UDP) is retained as a valuable benign source.

Change 3 — CTU-13 binetflow state mapping to Zeek equivalents
  Rationale: Production only sees Zeek states. CTU-13 Argus states are
  meaningless at inference time. Mapped in load_ctu13():
    INT        → S1   (mid-flow, established, no FIN)
    CON        → SF   (completed connection)
    FSPA_FSPA  → SF   (FIN bidirectional = completed)
    FSA_FSA    → SF
    PA_PA      → OTH  (PSH-ACK only, no SYN seen)
    S_         → S0   (SYN only)
    SRPA_SPA   → RSTO (RST from originator)

Change 4 — CTU-Normal cap: 80k → 100k
  Rationale: CTU-Normal is the only significant SF benign source (94% SF).
  Increasing its cap ensures SF benign patterns compete effectively against
  the remaining S0-dominated sources after IoT-23 reduction.

Change 5 — Drop CICIDS2017 entirely
  Rationale: Every CICIDS2017 sample has conn_state="-" and proto="unknown"
  — both critical discriminative features are absent on all 80k samples.
  CICFlowMeter does not produce Zeek-compatible features. Additionally,
  documented label accuracy issues (~10-15% mislabeled) add noise.
  UNSW-NB15 (Bro/Zeek-generated, proper conn_state) covers the attack
  diversity that CICIDS2017 was intended to provide.

Change 6 — 2:1 benign:attack ratio (120k attacks, 240k benign)
  Rationale: Real networks are overwhelmingly benign traffic. Training on a
  1:1 balanced dataset implicitly teaches the model to expect attacks half the
  time, making it trigger-happy. A 2:1 ratio is a conservative step toward
  realistic class distribution without sacrificing attack recall severely.
  The primary complaint about v6 in production is false positives; reducing
  them justifies some attack recall trade-off.

Change 7 — Source-stratified train/eval split
  Rationale: The v6 random 90/10 split produced an eval set that mirrored the
  training distribution. CTU-Normal failures (18% FP recall) were invisible
  during training because the eval set had the same IoT-23-dominated S0 bias.
  v7 holds out 10% from each (source, class) bucket independently, so eval_loss
  reflects performance across all sources from the first epoch.
