# v9.1 Plan: Behavioral Zeek Classification from Multi-Log Files

## Goal

Move beyond single-flow classification so the model can take Zeek logs from a
capture and classify flows using:

1. The flow itself (`conn.log`)
2. Optional application-layer context (`http.log`, `dns.log`, `ssl.log`, and later others)
3. Short behavioral summaries computed from neighboring Zeek flows
4. A second-pass host/window summary that can correct or reinforce per-flow decisions

The final user-facing direction remains:

- input: Zeek/Bro files from a capture
- output: per-flow verdicts (`ATTACK` or `FALSE POSITIVE`) and later optional host-level verdicts

`preprocess_zeek.py` remains the core dataset builder. The internet-sourced
Stratosphere / CTU datasets remain important because they provide:

- Zeek/Bro logs (`conn.log`, `dns.log`, `http.log`, `ssl.log`, etc.)
- labeled `.binetflow` files
- a reliable way to map conn.log flows to ground-truth attack/benign labels

The current CTU-Malware download + 5-tuple matching path should be extended, not replaced.

---

## v9.1 Priorities

### Priority 1: Behavioral window features

Problem:
`v9.0` is still mostly a single-flow classifier. It can miss behavior that only
becomes obvious across nearby flows: scans, beaconing, brute force bursts,
fan-out, repeated same-size connections, and low-rate persistence.

Plan:
For each conn.log row used in training or inference, compute compact behavioral
features from a short time window around that flow.

Recommended window views:

- same source host over last 60 seconds
- same source host over last 300 seconds
- same `(orig_h, resp_h)` pair over last 300 seconds

Recommended derived features:

- `src_conn_60s`
- `src_conn_300s`
- `src_unique_dst_60s`
- `src_unique_ports_60s`
- `src_s0_60s`
- `src_rsto_60s`
- `src_sf_60s`
- `pair_conn_300s`
- `pair_mean_gap_s`
- `pair_periodic_score`
- `same_resp_port_60s`
- `same_flow_size_repeats_300s`

These should be numeric and compact, not raw history dumps.

Why:

- port scans become visible via many short `S0` flows to many ports
- brute force becomes visible via repeated short attempts to one service
- beaconing becomes visible via repeated low-volume periodic flows
- P2P botnet behavior becomes visible via fan-out to many peers

---

### Priority 2: Hard benign negatives with context

Problem:
Multi-log context helps detection, but it also risks a shortcut:
"flow has HTTP/DNS/SSL section, therefore ATTACK."

Plan:
Add targeted benign samples that are superficially suspicious, especially from
real Zeek captures that also include app-layer logs.

Priority benign patterns to mine:

- short periodic HTTPS telemetry/update checks
- failed or reset TLS handshakes on port 443
- proxy traffic on `8080` / `3128`
- DNS NXDOMAIN and retry behavior from legitimate resolvers
- benign HTTP polling / health checks
- internal admin traffic to uncommon but legitimate management ports
- CDN / load balancer resets
- normal SMTP or relay traffic where present

Source strategy:

- keep existing CTU-Normal and UWF benign sources
- search CTU/Stratosphere captures with rich Bro logs and normal/background rows
- prefer sources that provide both conn.log and app-layer logs
- keep using binetflow labels where available

Hard-negative mining rule:

- do not only sample random benign flows
- explicitly oversample benign flows that look attack-like by port/state/size/context

Expected effect:

- fewer false positives on real enterprise traffic
- reduced context-presence shortcut
- stronger OOD generalization

---

### Priority 3: Host-level second pass

Problem:
Users ultimately provide Zeek files for a whole capture, not a single flow.
Some attacks are better recognized at host/window level than at isolated-flow level.

Plan:
Keep the existing per-flow classifier as pass 1, then add a second pass that
summarizes host behavior over a short window and produces:

- host-level behavioral summary
- optional refined verdicts for the host's suspicious flows
- optional host-level `ATTACK` / `FALSE POSITIVE` summary

Input to pass 2:

- pass-1 flow verdicts
- behavioral window features
- counts by state / port / protocol
- top suspicious destinations and services
- app-layer context summary counts

Output of pass 2:

- host summary report
- host attack likelihood
- optional per-flow override only when evidence is strong

This should begin as a reporting/re-ranking stage, not a destructive replacement
of pass-1 flow results.

## Data Strategy

### Internet-sourced labeled data remains central

The CTU/Stratosphere public dataset path should remain the backbone for `v9.1`.

Workflow:

1. Download capture files from `https://mcfp.felk.cvut.cz/publicDatasets/`
2. Pull Zeek/Bro logs from the capture's `bro/` directory
3. Pull labeled `.binetflow` file for the same capture
4. Match `conn.log` flows to `.binetflow` labels using normalized 5-tuple
5. Join app-layer logs by `uid`
6. Compute windowed behavioral features from the Zeek timeline
7. Build training samples from the enriched flow record

This keeps labels grounded in the original research data instead of inventing
heuristic labels locally.

### Additional capture selection for v9.1

Focus on captures that add:

- more HTTP C2 diversity
- more benign HTTP/TLS/DNS traffic with context
- more mixed-protocol botnets
- more repetitive bot behavior useful for temporal features

Candidate directions:

- more Zeus variants
- more Kelihos-like P2P / spam scenarios
- more benign-rich captures with app-layer logs

Botnet-3 (Kelihos) should remain an OOD hold-out unless there is a deliberate
decision to replace it with a different hold-out.

## Prompt Design for v9.1

### Pass 1: enriched per-flow prompt

Extend the current prompt with a compact `[BEHAVIOR]` section when available.

Example:

```text
[BEHAVIOR]
  Src Conns 60s:        42
  Unique Dst IPs 60s:   18
  Unique Dst Ports 60s: 14
  S0 / RSTO / SF 60s:   31 / 6 / 2
  Pair Conns 300s:      9
  Mean Gap:             29.8s
  Periodic:             High
  Same-Size Repeats:    8
```

This section should be omitted when the behavioral view cannot be computed.

### Pass 2: host/window prompt

Create a separate host-level prompt format summarizing:

- the host's recent flow distribution
- suspicious ports/services
- periodicity and repetition
- proportion of pass-1 attack predictions
- notable HTTP/DNS/SSL signals

This prompt should classify the host/window, not a single flow.

## Implementation Plan by File

### `preprocess_zeek.py`

Add:

- a reusable conn.log timeline loader that preserves timestamps and host keys
- behavioral feature computation per flow
- hard-benign mining hooks and sampling controls
- optional host/window training sample builder
- richer diagnostics:
  - context coverage by label
  - behavior feature coverage by label
  - hard-negative counts by source

Refactor idea:

- separate "load raw flows"
- separate "attach auxiliary context"
- separate "attach behavioral features"
- separate "sample / split / oversample"

### `prompt_utils.py`

Add:

- `behavior_ctx` support to `build_prompt()`
- helper formatting for `[BEHAVIOR]`
- optional separate builder for host-level prompts

### `classify_conn_log.py`

Add:

- timeline-aware behavior extraction during inference
- optional `--host-pass2` mode
- optional output of both pass-1 and pass-2 verdicts
- JSON output mode for downstream analysis

Potential CLI additions:

- `--behavior-window 60`
- `--behavior-window-long 300`
- `--host-pass2`
- `--json-out PATH`

### `benchmark_realworld.py`

Add:

- separate metrics for:
  - conn-only
  - conn + app-layer context
  - conn + app-layer + behavior
- explicit OOD family metrics
- hard-benign false-positive slices
- optional host-level metrics

This benchmark should become the place that proves whether the three v9.1
changes actually help.

### New module suggestion: `behavior_features.py`

Create a separate module for:

- sliding-window flow statistics
- periodicity heuristics
- repeated-size heuristics
- host/window aggregation

Reason:
Behavior logic will otherwise make `preprocess_zeek.py` and `classify_conn_log.py`
too hard to maintain.

### New module suggestion: `classify_capture.py`

Longer-term entrypoint that accepts a directory of Zeek logs and runs the full
pipeline:

- parse logs
- join context
- compute behavior
- classify flows
- optionally run host pass 2
- emit capture report

This better matches the end goal than `classify_conn_log.py` alone.

## Training Changes

### Data mix

Continue multi-source training, but explicitly track:

- benign with app-layer context
- attack with app-layer context
- benign with behavior signal
- attack with behavior signal

Avoid a dataset where:

- only attacks have context
- only attacks have strong behavioral summaries

### Sampling changes

Add targeted oversampling for:

- benign hard negatives
- attack samples where behavior matters more than single-flow features
- OOD-like patterns that currently fail on Kelihos-style traffic

### Masking / ablation

Keep context masking, but add experiments for:

- app-layer masking rate
- behavior-section masking rate
- conn-state masking interaction with behavior features

This should be benchmarked rather than fixed by intuition.

## Evaluation Plan

### Required comparisons

Run at least these ablations:

1. `v9.0 baseline`
2. `v9.1-a`: + behavior only
3. `v9.1-b`: + hard benign mining only
4. `v9.1-c`: + behavior + hard benign mining
5. `v9.1-d`: + host second pass

### Required metrics

- overall MCC
- per-source accuracy / recall
- Botnet-3 OOD MCC
- false positives on benign-with-context slices
- recall on periodic / beacon-like attack slices
- recall on scanning / brute-force slices

### Success criteria

Minimum:

- preserve or improve v8.1/v9.0 in-distribution MCC
- improve Botnet-3 OOD MCC
- reduce false positives on benign TLS/HTTP/DNS context traffic

Strong result:

- behavior features improve Botnet-3 and other mixed-protocol/OOD families
- host pass 2 improves low-signal attacks without materially increasing false positives

7. Add host-level pass 2 prompt + inference path
8. Add capture-level entrypoint once the above works

---

## Scope Guardrails

To keep `v9.1` tractable:

- do not add every Zeek log type at once
- do not replace binetflow-based labels with heuristic labels
- do not jump straight to graph models or sequence transformers
- keep behavior features compact and interpretable
- keep host pass 2 optional until it is validated

The immediate goal is not "full SOC platform." The immediate goal is a better
Zeek-native classifier that reasons over flow + context + local behavior.
